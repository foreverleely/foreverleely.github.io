接下来说一下热更新的另一个作用,”应用动态更新”
找到项目目录下的 Staging 的文件夹下的 　config.xml 文件,我们可以在内容最下面找到 cordova-hcp 添加的内容

首先这里的 URL 路径对应的是服务器下的 chcp.json 路径,这里需要手动修改为自己服务器更新应用的地址 (关于应用更新,我们只需要将项目下的第一级目录下的 www 文件夹整体放到服务器上即可),

上面是第一个需要注意的文件,下面这个也需要注意
项目路径下,第一级目录下的 www 文件夹下的 chcp.json (chcp.json 和 chcp.manifest这个在上面的操作中由 cordova 生成)

release 实际对应的是热更新保存时间 (热更新环境下,我们保存修改的时间),
content_url 对应的是服务器下,需要存放之前提及的 www 文件夹的路径(最直接来说就是上面提到的 Staging 的文件夹下的 　config.xml 文件中 URL 路径,去掉 /chcp.json 之后的部分)
release 和 content_url 都会随热更新保存而动态变化。

应用更新

了解和知晓以上内容后,我们再来谈谈应用更新的具体过程,
1.在已经修改了项目目录下的 Staging -> config.xml 文件下的 URL 路径之后;
2.我们可以通过一些途径安装到模拟器或者真机上后退出应用;
3.在热更新服务开启的条件下(在项目下 cordova build 并且 cordova-hcp server 后),我们修改第一级目录下 www 文件夹下应用的界面文件(修改应用界面),保存;
4.将项目第一级目录下的 www 文件夹(一定注意是第一级目录下的,项目下会有两个 www 文件夹 ,第一级目录下的才是最新修改的)整体拷贝到服务器上;
5.打开服务器上拷贝好的 www 文件夹下的 chcp.json 文件,修改其 content_url 为当前 www 文件夹的路径地址,
6.打开模拟器或者之前真机上的这个应用,大概几秒钟(视更新的内容多少来计算更新时间,一般都非常快)。
需要注意的就是修改应用时,需要在热更新条件下进行,其二,两个路径 config.xml 记得修改,服务器上的 chcp.json 的 content_url 记得修改。
