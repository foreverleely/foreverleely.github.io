优缺点

Delegate(代理,委托)

代理几乎是iOS开发中最常用的传值方式,很多原生控件都是用到了代理这种设计模式,比如UISearchbar,UICollectionView等。



优点,
1.减少代码的耦合性,将事物监听和处理相分离;
2.清晰的语法定义,减少维护成本,较强的代码可读性;
3.不需要创建第三方来监听事物和传输数据;
4.一个控制器可以实现多个代理,满足自定义开发需求,灵活性较高。

缺点,
1.实现委托的代码过程有点繁琐;
2.当实现跨层传值监听时将加大代码的耦合性,并且程序的层次结构将更加混乱;
3.当对多个对象同时传值相应的时候,委托的易用性将大大降低。

NotificationCenter(通知)

通知也是iOS中常用的一种传值响应方法,AVFoundation框架中的MPMoviePlayerController在监听播放状态改变,播放停止等事件时使用的都是NotificationCenter.

NSNotification采用的单例设计模式,当给一个通知中心注册一个 key 以后,那么无论在什么地方只要给通知中心发送一个这个 key 的消息,那么就实现了通信传值,注册的通知对象就会调用相应的方法。


优点,
1.使用简单,代码精简;
2.解决了同时向多个对象监听响应的问题;
3.传值方便快捷, Context 自身携带相应的内容。

缺点,
1.使用完毕后要时刻记住注销通知,否则将出现不可预测的 crash ;
2.key 不够安全,编译器不会监测是否被通知中心正确处理;
3.调试的时候动作的跟踪很难进行;
4.当使用者向通知中心发送通知的时候,并不能获得任何反馈信息;
5.需要一个第三方的对象来做监听者与被监听者的中介。

Block(代码块)

Block 是iOS4.0+ 和 Mac OS X 10.6+ 引入的对C语言的扩展,用来实现匿名函数的特性。

闭包是一个能够访问其他函数内部变量的函数。

官方文档中也提到集中使用 Block 的场合,比如,
任务完成时回调处理;
消息监听回调处理;
错误回调处理;
枚举处理;
视图动画,变换;
排序。



优点,
1.语法简洁,实现回调不需要显示调用方法,代码更加紧凑;
2.增强代码的可读性和可维护性;
3.配合 GCD 优秀的解决多线程问题。

缺点,
1.Block 中的代码自动进行一次 retain 操作,容易造成内存泄漏;
2.Block 内默认引用为强引用,容易造成循环引用。

注意事项

代理

在代理中调用方法的时候用的是系统的子线程,因此,当使用 Delegate 进行 UI 操作的时候,必须调用 GCD 的主线程方法,

通知

在通知的使用过程中 crash 的原因多数是注册观察者而没有及时注销观察者。及时注销观察者很重要。

由于通知的使用比较简单,往往会出现很多开发人员乱用通知,导致代码的可维护性和可读性很差,即便是使用宏定义也不能完全避免这个问题,当观察者比较多时,就很容易扰乱整个代码,在这方面, Block 和 Delegate 比通知达到的效果就要好很多。

Block

当在 Block 中引用某个外部变量的时候, Block 内部只会进行拷贝,也就是说,即便你在 Block 之前改变了那个外部变量,你在 Block 里那个外部变量的值还是最开始的那个外部变量的值。
同步外部变量的值,只需要在 Block 内部引用变量时,在前面加上 __block 关键字。

使用场景

回调方法

在日常的开发过程中,我们经常会遇到一些完成之后的处理,比如完成某个网络请求之后的回调,或者页面加载完成以后的回调等,这个时候,一般使用 Block 或者 Delegate。

在一对一传输回调的时候 Block 明显更加简单高效。

在一对多的情况下, Delegate 比较好。

跨层通信

有的时候需要实现两个豪无关联的对象之间的通信,这个时候如果使用 Block 或者 Delegate 势必会增加代码的耦合性,影响代码的复用,这个时候使用 Notification 比较好。

UI响应事件

用户在于 App 的 UI 进行交互的时候,总需要 App 进行交互响应,苹果官方的建议是使用代理。

简单传值

当需要进行简单传值的时候,一般而言, Block 还是首选。
